01LLM 내부 ①

토큰화 → 임베딩 → 의미 공간

문장을 조각(토큰)으로 나누고, 각 조각을 의미를 담은 좌표(벡터)로 변환합니다. 이때 비슷한 의미를 가진 단어들은 가까운 위치에 모이게 됩니다.

① 토큰화 — 텍스트를 조각내고 고유 번호 부여

인공#8123 지능#4502 은#11 똑똑#9981 하다#205

"인공지능은 똑똑하다" → ID 배열 [8123, 4502, 11, 9981, 205]

② 임베딩 — 각 토큰을 의미를 가진 벡터로 변환

지능 #4502 ▶

실제로는 768개에서 수천 개의 숫자로 이루어져 있으며, 이 숫자들의 조합이 곧 단어의 '의미 좌표'가 됩니다.

👑 왕 👑 여왕 🍎 사과 🍌 바나나 🐱 고양이 🐶 강아지
'왕'과 '여왕'은 가까운 곳에, '사과'와 '바나나'는 다른 곳에 모입니다. 이를 통해 AI는 단어 간의 유사성을 이해하며, '왕 - 남자 + 여자 = 여왕' 같은 의미 연산도 가능해집니다.

④ 어텐션(Attention) — 문맥상 어디에 주목해야 할지 결정하는 핵심 기술

색이 밝을수록 더 주목한다는 뜻입니다. 예를 들어 '찾았다'라는 단어는 '돈을'과 '은행'이라는 단어에 주목하여 문맥을 파악합니다.

⑤ 트랜스포머 층

여러 층을 거치며 단어의 의미와 문맥이 점점 정교하게 분석됩니다.

⑥ 출력 (Softmax)

다음에 올 단어들의 확률을 계산하여 가장 적절한 단어를 선택합니다.

똑똑하다64%
유용하다21%
빠르다11%

③ 위치 인코딩 — 단어의 순서 정보 기억

"개가 pos1 사람을 pos2 물다 pos3"와 "사람이 개를 물다"는 단어는 같지만 위치가 달라 의미가 정반대입니다. 그래서 AI는 각 단어 조각(토큰)에 순서 정보를 함께 입력받습니다.

④ 트랜스포머 구조의 핵심인 '어텐션'은 특정 단어가 문장 내의 다른 단어들과 얼마나 연관되어 있는지 계산하여 전체적인 문맥을 이해하게 해줍니다.

⚠️ 사용자의 질문에 답변한다는 개념이 없고, 단순히 입력된 문장에 이어지는 글을 작성합니다.

② 지도 미세조정(SFT) 후🧑‍🏫 사람이 작성한 좋은 질문과 답변 예시를 학습하여 모방하는 단계

🤖 "안녕하세요. 저는 AI 언어 모델이므로 감정을 가지고 있지 않습니다. 도움이 필요하시면 알려주세요."

✓ 답변하는 형식은 갖췄지만, 어투가 딱딱하고 사무적입니다.

③ 인간 피드백(RLHF) 후👍 사람이 긍정적으로 평가한 답변 스타일을 학습하여 강화하는 단계

🤖 "고마워요, 그렇게 물어봐 주셔서 😊 저는 감정은 없지만 오늘도 도움이 될 준비는 됐어요! 본인 기분은 어떠세요?"

⭐ 말투가 자연스럽고 친절해지며, 사용자에게 되묻기도 합니다. 우리가 현재 사용하는 AI 모델들은 대부분 이 3단계까지 학습을 마친 상태입니다.

06API 데이터 흐름

08구조 학습

계속해서 코드를 '다시 만드는' 시행착오를 줄이려면 — 개발 전 먼저 '구조'부터 학습해야 합니다.

머릿속에 있는 아이디어만으로 무작정 코드를 짜라고 지시하면, 기초 설계가 부실해져 결국 처음부터 다시 만들어야 하는 상황이 생깁니다.

❌ 바로 시키기

🙋 "예약 시스템 만들어줘"
🤖 (AI가 뼈대 없이 임의로 구현함) → 에러 발생 → 코드 수정 → 결제 기능 추가 실패 → 또 수정 → 알림 기능 연동 실패 → 끝없는 재작업...

→ 기초 토대가 부실하여 기능이 추가될 때마다 코드를 완전히 새로 짜야 합니다.

✅ 구조 먼저 학습 후 구현

🙋 '예약 시스템을 만들려면 어떤 기능들이 필요하고, 전체적인 데이터 흐름과 시스템 구조는 어떻게 잡아야 할까?'
🤖 시스템 구성 요소: ①사용자 인증, ②예약 데이터베이스, ③결제 시스템, ④알림 전송, ⑤관리자 페이지. 전체 흐름: 예약 요청 → 빈자리 확인 → 결제 진행 → 예약 확정 → 안내 알림 발송.

→ 이렇게 탄탄한 토대를 먼저 세우고 나면, 필요한 부분만 수정하거나 기능을 쉽게 확장할 수 있습니다.

'학습 → 설계 → 구현' 순서로 진행하면 불필요한 재작업 굴레에서 벗어날 수 있습니다. AI는 코딩 도구일 뿐만 아니라, 시스템 구조를 가르쳐주는 훌륭한 선생님이기도 합니다.

전망UI · UX · AX

AI 시대를 맞아 소프트웨어 인터페이스의 패러다임이 UI에서 UX로, 그리고 이제는 AX로 진화하고 있습니다.

사용자가 화면을 일일이 클릭하며 조작하던 시대(UI)를 지나, 편리한 사용 경험을 정교하게 설계하는 시대(UX)를 거쳐, 이제는 AI 에이전트가 사람의 의도를 파악해 대신 일해주는 새로운 차원의 시대(AX)로 넘어가고 있습니다.

AX

Agent Experience (에이전트 경험) · '이제는 AI가 알아서 대신 처리한다'
🤖사용자는 최종 목표만 툭 던져주고, AI 에이전트가 내부적으로 도구를 활용해 스스로 작업을 수행하고 결과만 보고함
🏗️겉으로 보이는 화려한 화면 디자인보다 AI가 데이터를 잘 이해하고 소통할 수 있는 뒷단 시스템의 구조와 명세(스키마, MCP 등)가 훨씬 중요한 핵심 요소로 떠오름
🤝소프트웨어를 '내가 직접 다루는 편리한 도구'로 보는 것을 넘어, '나와 협업하는 똑똑한 동료'로 경험하게 만드는 설계
💡 무엇이 바뀌나

앞으로의 시스템 개발은 두 가지 방향으로 나뉩니다. 사람이 보는 화면은 극도로 직관적이고 심플하게 만들되, 그 보이지 않는 뒷단은 AI 에이전트가 완벽하게 읽고 조작할 수 있도록 매우 구조적·의미적이고 체계적인 형태로 구축해야 합니다. 즉, 사람을 위한 심플함과 AI를 위한 정교함이 모두 필수적인 시대입니다.

📊 왜 지금

리서치 기관의 전망에 따르면, 2026년에는 글로벌 기업용 애플리케이션의 약 40% 이상이 업무용 AI 에이전트를 기본으로 탑재할 것 으로 예상됩니다(2025년 기준 5% 미만). 시스템에 얼마나 훌륭한 AX(에이전트 경험)를 녹여내느냐가 곧 기업 솔루션의 새로운 경쟁력이 될 것입니다.

💡 세 줄 요약: 과거의 UI가 '쓰기 편한 망치를 예쁘게 만드는 것'이고, UX가 '못을 박는 경험을 수월하게 설계하는 것'이었다면, 미래의 AX는 '망치질을 대신해 줄 듬직한 AI 동료가 가장 효율적으로 일할 수 있는 환경을 만들어 주는 것'이라고 정의할 수 있습니다.

심화스키마 (Schema)

스키마(Schema)의 개념 — 데이터가 반드시 따라야 하는 깐깐한 '구조적 규칙서' 혹은 '설계도'

입력받을 데이터 항목(필드)이 무엇인지, 각 항목이 어떤 형식(타입)이어야 하는지, 그리고 반드시 필요한 값인지(필수/선택)를 사전에 엄격히 정의해 둔 형틀입니다. AI의 출력을 이 스키마에 맞추도록 강제하면 엉뚱한 대답(환각)이나 시스템 에러를 유발하는 형식 오류를 원천적으로 차단할 수 있습니다.

Fig 1. Schema 정의
contact_schema = { name : string (필수) // 이름 phone : string (필수) // 010-####-#### amount: number (필수) // ≥ 0 date : string (선택) // YYYY-MM-DD }
Fig 2. 인스턴스 검증 (validate)
✓ 적합{ name:"김철수", phone:"010-1234-5678", amount:12000 }
✗ 위반{ name:"김철수", phone:"전화없음", amount:"공짜" } └ 실패 사유: phone 필드에 전화번호 형식이 아닌 텍스트가 들어갔고, amount 필드에 숫자 타입이 아닌 문자열('공짜')이 들어가 스키마 규칙을 심각하게 위반했습니다.
Fig 3. AI에서 스키마의 3가지 역할
📐 출력 강제"'반드시 이 JSON 스키마 구조로만 대답해'라고 지시하면 → AI가 정해진 형틀에 정확히 들어맞는 깔끔한 구조화 데이터를 출력합니다."
🛡️ 검증 게이트AI가 스키마에 어긋난 엉터리 형식으로 답변을 내놓으면, 시스템 게이트가 이를 즉시 거부하고 AI에게 에러 메시지를 던져 재작업을 강제합니다. 즉, 불량품은 절대 다음 단계로 넘어가지 못합니다.
🏗️ 구조 재설계AI가 잦은 실수를 저지른다면 프롬프트를 고칠 게 아니라, 헷갈리기 쉬운 스키마의 구조 자체를 명확하게 재설계하여 실수가 일어날 여지 자체를 없애버려야 합니다.
💡 스키마 파트 핵심 요약: 시스템 세상에서 스키마란 데이터의 깐깐한 '계약서(Contract)'입니다. 모호한 줄글 텍스트 대신, 톱니바퀴처럼 꽉 짜인 스키마 구조로 데이터를 주고받게 만들면 AI, 개발자, 그리고 시스템 프로그램 모두가 오류 없이 동일한 형식으로 소통할 수 있는 탄탄한 자동화의 토대가 마련됩니다.

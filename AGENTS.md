# AI 강의 자료 저장소 작업 규칙

이 문서는 Codex, Claude Code 등 저장소에서 작업하는 AI가 반드시 따라야 하는 기본 규칙이다. 사용자의 최신 지시가 이 문서보다 우선하지만, 명시적으로 변경되지 않은 항목은 계속 유지한다.

## 1. 작업 시작 전에 먼저 확정할 것

작업을 시작하기 전에 아래 내용을 5줄 이내로 정리한다.

1. 이번 작업의 목적
2. 수정할 원본 파일
3. 함께 갱신할 산출물
4. 변경하지 않을 범위
5. 완료 판정 기준

정보가 부족하면 한 번에 최대 3개만 질문한다. 결과가 크게 달라지지 않는 사소한 항목은 기존 스타일과 이 문서의 기본값을 사용한다.

새 콘텐츠 요청은 구현 전에 기존 목차를 검색하고 `추가`, `대체`, `통합` 중 하나로 분류한다. 같은 주장을 다른 말로 반복하는 슬라이드를 만들지 않는다.

## 2. 원본과 산출물

### 원본

- 공개 강의 HTML의 원본은 `slides.html`이다.
- 강의 문구, 슬라이드 순서, UI 예시는 우선 `slides.html`에서 수정한다.
- PDF 파일을 직접 수정해 원본과 다른 내용을 만들지 않는다.
- PDF 생성 도구를 사용할 때는 입력 파일, 화면 크기, 폰트, 워터마크, 실행 명령을 재현 가능하게 기록한다.
- 임시 로컬 도구를 정식 생성 도구로 간주하지 않는다. 반복 사용할 도구는 검증 후 추적 여부를 결정한다.

### 공개 산출물

- PC PDF: `downloads/ai-automation-vibe-coding-moyi.pdf`
- 모바일 PDF: `downloads/ai-automation-vibe-coding-moyi-mobile.pdf`
- PC·모바일 선택 페이지: `downloads/index.html`

슬라이드 내용이 바뀌면 특별한 제외 지시가 없는 한 HTML, PC PDF, 모바일 PDF, 선택 페이지의 페이지 수 안내를 함께 갱신한다.

## 3. 디자인 기본값

- 기존 어두운 배경, 파란색·보라색·민트 강조색과 타이포그래피를 유지한다.
- 사용자가 요구하지 않은 통계, 차트, 대시보드 숫자를 임의로 추가하지 않는다.
- 한 슬라이드는 하나의 핵심 주장만 전달한다.
- 설명보다 실제 업무 예시, 입력 화면, 처리 과정, 검증 결과를 우선한다.
- 문장은 발표자가 실제로 말할 수 있는 자연스러운 한국어로 작성한다.
- 청중의 기본값은 사내 비개발자다. 행동을 나타내는 쉬운 한국어를 먼저 쓰고 전문용어는 필요한 경우에만 병기한다.
- 새 개념에는 가능한 한 실제 입력, 처리 과정, 결과 또는 실패 사례를 연결한다.

## 4. 모바일 제작 규칙

- 모바일 페이지는 9:16 세로형으로 만든다.
- PC 슬라이드를 통째로 축소하거나 기계적으로 좌우 절반으로 자르지 않는다.
- 제목·본문·UI·표를 의미 단위 카드로 재배치한다.
- 확대하지 않고 읽을 수 있어야 한다.
- 권장 크기: 제목 52~68px, 본문 30~36px, 보조 문구 최소 24px.
- 긴 페이지는 문장, 카드 또는 섹션 사이의 여백에서 나눈다. 문장과 UI 카드 중간에서 자르지 않는다.
- 애니메이션은 한 페이지에 한 컷을 원칙으로 한다.
- 모바일 워터마크는 콘텐츠 중앙에 배치하지 않는다.
- 워터마크 기본값은 하단 전용 여백의 작은 `@MOYI · MOBILE EDITION`이다.

## 5. 루프와 오케스트레이션 설명 기준

- 루프: 한 작업 안에서 `실행 → 검증 → 수정·재시도 → 예외 전달 → 완료`가 반복되는 구조.
- 오케스트레이션: 여러 작업자와 도구 사이의 역할, 순서, 데이터 전달, 실패 처리와 승인 지점을 조정하는 구조.
- 둘을 같은 의미로 사용하지 않는다.
- 모든 루프에는 완료 조건, 재시도 한도, 사람 승인 또는 예외 전달 기준을 명시한다.
- 모든 오케스트레이션에는 각 역할의 입력, 출력, 완료 조건과 로그 책임을 명시한다.

## 6. 작업 누락 방지 체크리스트

콘텐츠 변경 시 아래 표를 확인한다.

| 항목 | 확인 내용 |
|---|---|
| HTML | 새 문구와 슬라이드가 올바른 위치에 있는가 |
| PC PDF | HTML과 같은 내용이며 페이지가 잘리지 않는가 |
| 모바일 PDF | 확대 없이 읽히며 의미 단위로 나뉘는가 |
| 선택 페이지 | 실제 페이지 수와 설명이 일치하는가 |
| 링크 | 기존 공개 URL이 유지되는가 |
| 워터마크 | 콘텐츠를 가리지 않는가 |
| 문구 파일 | 사용자가 별도 수정할 전체 문구 파일이 최신인가 |
| 검수 | 첫 장, 변경 장, 정보량이 많은 장, 마지막 장을 확인했는가 |

## 7. Git 작업 규칙

- 작업 전 `git status -sb`와 최근 커밋을 확인한다.
- `tmp/`, 렌더링 이미지, QA 이미지, 생성 스크립트를 공개 산출물과 함께 무심코 커밋하지 않는다.
- 관련 파일만 경로를 명시해 스테이징한다. `git add -A`를 기본값으로 사용하지 않는다.
- 하나의 커밋에는 하나의 사용자 목적만 담는다.
- 콘텐츠 추가, PDF 재생성, 배포 안내 수정이 하나의 기능을 완성한다면 같은 커밋에 포함한다.
- 커밋 전에 `git diff --cached --stat`로 포함 파일을 확인한다.
- 커밋 후 원격 브랜치와 동기화 상태를 확인한다.
- 공개 배포 후 HTTP 200, 새 파일 크기, 핵심 문구 반영 여부를 확인한다.
- 작업 전 저장소의 Git 작성자 이름과 이메일을 확인하고 기존 기준과 다르면 보고한다.
- 기준이나 생성 방식이 바뀌면 `docs/DECISIONS.md`와 관련 체크리스트를 같은 커밋에서 갱신한다.

권장 커밋 메시지 예시:

- `Add loop and orchestration concepts`
- `Rebuild mobile PDF for readability`
- `Refine mobile watermark placement`

## 8. 브라우저 동작 검수

- 슬라이드의 첫 진입, 이전·다음 이동, 키보드·클릭·스와이프를 확인한다.
- 애니메이션은 코드 존재가 아니라 실제 시작, 재진입, 반복 또는 종료 상태로 판정한다.
- 새 UI는 빈 상태, 처리 중, 성공, 검토 필요, 실패 상태를 확인한다.
- 요청하지 않은 숫자와 통계가 실제 데이터처럼 보이지 않는지 확인한다.

## 9. 컨텍스트 압축·새 세션 인수인계

- 압축이나 새 세션 전에 목표, 완료·미완료, 확정 결정, 수정 파일, 검증 결과, 실패한 방법과 다음 행동을 `CURRENT_TASK.md`에 기록한다.
- 영구 규칙은 `AGENTS.md`, 확정 결정은 `docs/DECISIONS.md`, 현재 작업 상태는 `CURRENT_TASK.md`로 분리한다.
- 새 세션에서는 문서와 `git status`, 최근 관련 커밋을 먼저 읽고 현재 상태를 요약한 뒤 수정한다.
- 인수인계 문서의 내용이 실제 파일과 Git 상태에 맞는지 확인한다.
- 템플릿은 `docs/CURRENT_TASK_TEMPLATE.md`를 사용한다.

## 10. 완료 보고 기준

완료 보고에는 다음 내용만 명확히 적는다.

- 무엇을 변경했는지
- PC·모바일 페이지 수
- 공개 링크
- 검수한 항목
- 남아 있는 제한이나 사용자가 결정해야 할 사항

로컬 파일 경로만 제공하고 외부 공개 링크처럼 설명하지 않는다. 공개 링크가 필요한 요청은 실제 외부 접속까지 확인해야 완료다.

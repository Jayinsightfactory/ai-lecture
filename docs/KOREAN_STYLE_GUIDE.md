# 한국어 문장 작성 기준

이 문서는 `slides.html`, `index.html`, 다운로드 안내와 PDF에 들어가는 한국어를 자연스럽게 유지하기 위한 고정 기준이다. 새 문장을 쓰거나 기존 문장을 고칠 때 반드시 적용한다.

## 1. 기본 화법

- 독자는 사내 비개발자다. 옆자리 동료에게 설명하듯 쓴다.
- 제목은 결론부터 짧게 쓴다. 본문은 `-습니다`체로 통일한다.
- 카드 제목은 명사형으로 짧게, 카드 설명은 완전한 문장으로 쓴다.
- 한 문장에는 메시지 하나만 담는다. 쉼표가 세 번 이상 나오면 문장을 나눈다.
- 번역투보다 한국어에서 실제로 쓰는 동사를 고른다. `수행하다`보다 `처리하다`, `확인하다`, `만들다`, `연결하다`를 우선한다.
- 주어가 분명하면 능동형으로 쓴다. `결과가 생성됩니다`보다 `AI가 결과를 만듭니다`가 자연스럽다.
- 같은 문단에서 `~할 수 있습니다`, `~해야 합니다`, `~됩니다`를 반복하지 않는다.

## 2. 피해야 할 표현

| 피할 표현 | 권장 표현 |
|---|---|
| 업무 표면이 늘어났습니다 | AI를 쓰는 화면과 도구가 늘었습니다 |
| 작업을 끝까지 수행합니다 | 중간에 놓치지 않고 마무리합니다 |
| 실행 조합입니다 | AI와 도구를 함께 쓰는 방식입니다 |
| 업무 계약을 고정합니다 | 바뀌지 않는 업무 기준을 정합니다 |
| 역량을 극대화합니다 | 필요한 일을 더 잘 처리합니다 |
| 기능을 제공하고 있습니다 | 이 기능으로 ○○할 수 있습니다 |
| ~에 가깝습니다 | 실제로 달라진 점은 ○○입니다 |
| 비로소 거듭납니다 | 실제로 일을 처리하는 에이전트가 됩니다 |
| 원천적으로 차단합니다 | 처음부터 막습니다 |
| 즉답을 줄 수 있어야 합니다 | 화면만 보고 바로 알 수 있어야 합니다 |

## 3. 제품과 기능을 설명하는 순서

제품명을 먼저 나열하지 않는다. 아래 순서로 쓴다.

1. 무엇을 할 수 있는지
2. 어떤 상황에서 쓰는지
3. 결과로 무엇이 남는지
4. 필요할 때만 제품명과 영문명을 덧붙이기

예시:

- 나쁨: `Artifacts는 인터랙티브 결과물을 제공합니다.`
- 좋음: `대화 중에 만든 화면이나 그래프를 바로 확인하고 고칠 수 있습니다. Claude에서는 이 기능을 Artifacts라고 부릅니다.`

## 4. 슬라이드 제목 기준

- 제목만 읽어도 결론이 보여야 한다.
- `~에 대한 이해`, `~의 개념`, `~의 고찰` 같은 보고서식 제목을 피한다.
- 가능하면 35자 안팎으로 쓴다.
- 과장된 단정과 광고 문구를 피한다.

예시:

- 나쁨: `AI 플랫폼 확장에 따른 업무 환경의 패러다임 변화`
- 좋음: `이제 AI는 대화창 밖에서도 일합니다`

## 5. 용어 표기

- 제품명은 공식 표기를 유지한다: `Claude Code`, `Codex`, `Firebase Studio`, `NotebookLM`.
- 전문용어는 처음 한 번만 한글 설명과 영문을 함께 쓴다.
- 이미 널리 쓰는 `API`, `PDF`, `Excel`, `Git`은 억지로 번역하지 않는다.
- `툴`과 `도구`가 섞이면 본문에서는 `도구`, 공식 제품명에서는 원문을 쓴다.
- `AI가 알아서`처럼 책임 범위가 모호한 표현 뒤에는 검증·승인 조건을 붙인다.

## 6. 최종 검수 질문

- 이 문장을 실제 회의에서 그대로 말해도 어색하지 않은가?
- 번역한 영어 문장처럼 들리는 명사 나열이 없는가?
- 누가 무엇을 하는지 바로 알 수 있는가?
- 제품명보다 사용자가 얻는 결과가 먼저 보이는가?
- 한 문장에 같은 뜻이 반복되지 않는가?
- 소리 내어 읽었을 때 숨이 차지 않는가?

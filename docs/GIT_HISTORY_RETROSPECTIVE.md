# 44개 커밋으로 본 작업 방식 회고

## 분석 범위와 한계

- 대상: 2026-05-24부터 2026-07-17까지의 전체 44개 커밋
- 규모: 15,431줄 추가, 5,699줄 삭제
- 작업 밀도: 26개 커밋이 직전 커밋 후 15분 이내, 33개가 1시간 이내에 생성됨
- 확인 자료: 커밋 시각, 메시지, 변경 파일, diff와 현재 저장소 구조

Git에는 당시 대화의 질문과 답변 전문이 저장되어 있지 않다. 따라서 정확한 문장을 복원한 것이 아니라, 어떤 요구가 언제 추가됐고 어떤 수정이 반복됐는지를 변경 기록으로 추론했다.

## 결론

이 저장소의 문제는 작업 속도가 느린 것이 아니라 **빠르게 추가하는 속도에 비해 전체 구조를 다시 확인하는 장치가 늦게 생긴 것**이다. 결과물은 계속 좋아졌지만 다음 흐름이 반복됐다.

`새 아이디어 추가 → 비슷한 설명 누적 → 뒤늦은 중복 정리 → 시각화 교체 → 동작 오류 수정 → 다른 형식으로 다시 생성`

초기에 콘텐츠 지도, 대상 사용자, 산출물 관계, 모바일 기준, 완료 조건을 문서화했다면 상당수 재작업을 줄일 수 있었다.

## 반복 문제와 근거

### 1. 추가가 구조 설계보다 먼저였다

초기 13개 커밋은 대부분 한 파일에 새 섹션과 예시를 더했다. 이후 `df3a77a`, `a1b4db4`에서 중복 제거와 전면 정리가 이뤄졌다. 새 요청마다 바로 구현하기 전에 “추가·대체·통합 중 무엇인지”를 판단하는 단계가 필요했다.

> 새 슬라이드를 만들기 전에 현재 목차에서 같은 주장을 하는 슬라이드를 검색하고, 추가·대체·통합 중 하나를 선언한다.

### 2. 실제 사례 우선 기준이 늦게 확정됐다

`313dda4`에서 텍스트 카드를 실제 대화창으로 바꾸고, `6b85361`에서 실제 Claude 작업 사례를 추가했다. 강의의 핵심 기준을 처음부터 “설명보다 실제 입력·처리·결과”로 뒀다면 앞선 카드 교체를 줄일 수 있었다.

> 개념 슬라이드마다 최소 하나의 실제 입력, 처리 흔적, 결과 또는 실패 예시를 연결한다.

### 3. 청중 수준이 뒤늦게 반영됐다

`0fa0d48`에서 Grep, Read, Edit, Verify 같은 용어를 초보자용 한국어로 바꿨다. 콘텐츠 작성 전에 “사내 비개발자”라는 대상과 허용할 전문용어 수준을 명시했어야 한다.

> 전문용어는 한국어 행동명을 먼저 쓰고, 필요한 경우에만 영문을 괄호로 병기한다.

### 4. 애니메이션을 ‘있음’으로 확인하고 ‘작동함’으로 검증하지 않았다

`80700fb` 이후 `e142544`, `0a3e746`에서 CSS 정지 환경, JS 구동, TDZ 버그, 상태 머신을 연속 수정했다. 코드 존재 여부가 아니라 실제 브라우저에서 시작·반복·재진입·종료가 되는지를 검수해야 했다.

> 애니메이션은 첫 진입, 이전·다음 이동 후 재진입, 저성능 환경, 종료 상태를 실제 브라우저에서 확인한다.

### 5. 원본과 파생 산출물의 관계가 늦게 생겼다

`index.html`이 먼저 커졌고 `e88f200`에서 `slides.html`이 별도로 추가됐다. 7월에는 PDF 두 개와 선택 페이지가 더해졌다. 현재 저장소에는 PDF 생성 과정을 재현하는 추적된 스크립트나 명령 문서가 없어 다른 작업자가 같은 결과를 만들기 어렵다.

> 원본은 하나로 선언하고, PDF 생성 명령·폰트·화면 크기·워터마크 설정을 저장소에 추적한다. 생성 파일은 직접 편집하지 않는다.

### 6. 모바일을 별도 편집본으로 정의하지 않았다

`7124415`, `daa4a85`, `9f4fefd`, `2c4994f`는 모바일판과 워터마크가 요구를 보면서 단계적으로 정의됐음을 보여준다.

> 모바일은 1080×1920 9:16 재편집본이며, 확대 없이 읽히고 콘텐츠 카드 중간에서 자르지 않는다는 기준을 작업 전에 고정한다.

### 7. 저장소 운영 기준이 결과물보다 늦게 추가됐다

44번째 커밋에서 처음으로 `AGENTS.md`, 결정 기록, 체크리스트가 생겼다. 그전에는 원본, 산출물, 공개 URL, 커밋 범위를 매 작업마다 다시 추론해야 했다.

> 작업 시작 시 `README.md` → `AGENTS.md` → `docs/DECISIONS.md` → 체크리스트 순서로 읽고, 기준 변경은 코드와 같은 커밋에 기록한다.

## 앞으로의 권장 작업 순서

1. 요청을 “목적, 대상, 추가·대체·통합, 산출물, 제외 범위, 완료 조건”으로 정규화한다.
2. 현재 목차와 결정을 검색해 충돌과 중복을 먼저 보고한다.
3. HTML 한 곳에서 작은 단위로 수정한다.
4. 브라우저에서 내용, 내비게이션, 애니메이션을 검증한다.
5. 확정된 원본에서 PC·모바일 산출물을 생성한다.
6. 변경된 모든 유형의 페이지를 확인한다.
7. 관련 파일만 커밋하고 공개 URL에서 최종 결과를 확인한다.

## 우선순위

1. PDF 생성·검수 절차를 재현 가능한 스크립트와 문서로 저장
2. 슬라이드별 ID, 목적, 실제 사례를 관리하는 콘텐츠 목차 작성
3. HTML·PC·모바일·선택 페이지 동기화 자동 점검
4. 임시 폴더와 공개 도구의 커밋 정책 확정
5. Git 작성자 이름과 이메일을 저장소 기준으로 통일

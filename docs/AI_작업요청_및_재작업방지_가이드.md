# AI 작업 요청 및 재작업 방지 가이드

## 이 문서의 목적

이번 강의 자료 작업에서는 결과물이 점점 좋아졌지만, 모바일 PDF를 여러 번 다시 만들고 워터마크와 공개 링크를 뒤늦게 수정해야 했다. 문제는 AI의 작업 능력보다 작업 전에 확정하지 않은 조건과 여러 산출물의 동기화 방식에 있었다.

이 가이드는 다음 문제를 줄이기 위한 것이다.

- 같은 요구를 여러 번 설명하는 문제
- HTML은 수정됐지만 PDF에는 누락되는 문제
- PC와 모바일 버전의 내용이 달라지는 문제
- 파일은 만들어졌지만 공개 링크에서 보이지 않는 문제
- 임시 파일이나 관련 없는 변경이 커밋에 섞이는 문제
- 글씨, 워터마크, 페이지 비율을 결과를 본 뒤 다시 정하는 문제

## 1. 이번 작업에서 반복된 문제와 예방 문장

### 문제 1. “모바일 버전”의 의미가 불명확했다

처음에는 PC 페이지를 분할하거나 세로 화면 안에 축소했다. 결과적으로 확대가 필요하거나 내용이 의미 없이 잘렸다.

더 좋은 요청:

> 모바일판은 1080×1920의 9:16 세로형으로 만들어줘. PC 슬라이드를 축소하거나 좌우로 자르지 말고, 제목·본문·UI를 의미 단위 카드로 다시 배치해줘. 확대 없이 읽혀야 하고 본문은 최소 30px로 해줘.

### 문제 2. 워터마크의 목적과 위치가 늦게 정해졌다

“크게 넣어줘”만 지정하면 콘텐츠 중앙을 가리는 결과가 나올 수 있다.

더 좋은 요청:

> PC판은 중앙 대각선 워터마크를 10% 내외로 넣고, 모바일판은 콘텐츠와 겹치지 않는 하단 여백에 작은 서명형 워터마크를 넣어줘.

### 문제 3. “링크”가 로컬 링크인지 공개 링크인지 구분되지 않았다

로컬 파일 링크는 다른 사람이 열 수 없다.

더 좋은 요청:

> 작업이 끝나면 `jayinsightfactory.github.io/ai-lecture`에 공개 배포하고, 다른 기기와 로그아웃 상태에서도 열리는 URL을 확인해줘.

### 문제 4. 한 부분을 고쳤을 때 관련 산출물이 함께 갱신되지 않을 수 있었다

강의는 HTML, PC PDF, 모바일 PDF, 선택 페이지로 구성되어 있다. 하나만 수정하면 버전이 어긋난다.

더 좋은 요청:

> 강의 내용이 바뀌면 HTML, PC PDF, 모바일 PDF, 선택 페이지의 페이지 수와 설명을 모두 같은 커밋에서 갱신해줘.

### 문제 5. 사용자가 원하지 않은 요소가 추가될 수 있었다

업무 UI를 요청했는데 통계 화면처럼 해석될 가능성이 있었다.

더 좋은 요청:

> 통계 대시보드는 추가하지 마. 화면의 중심은 업로드, 처리 단계, 매칭, 검증, 미리보기, 다운로드, 작업 이력과 로그야.

## 2. 처음 요청할 때 사용하는 마스터 프롬프트

아래 빈칸을 채워 그대로 사용할 수 있다.

```text
[작업 목적]
내가 만들고 싶은 것은 ________________________________이다.

[사용자]
주 사용자는 __________________이고, 사용 환경은 PC / 모바일 / 둘 다이다.

[반드시 포함할 기능 또는 내용]
1. ________________________________
2. ________________________________
3. ________________________________

[포함하지 않을 것]
통계, 차트, 자동 발송 등 요청하지 않은 기능은 임의로 추가하지 않는다.
추가로 제외할 내용: ________________________________

[원본과 산출물]
원본 파일: ________________________________
최종 산출물: HTML / PC PDF / 모바일 PDF / PPT / 기타 __________

[PC 기준]
16:9 가로형이며 발표 화면에서 읽기 쉬워야 한다.

[모바일 기준]
9:16 세로형, 1080×1920 기준이다.
확대 없이 읽혀야 하며 본문은 최소 30px로 한다.
PC 화면을 단순 축소하거나 기계적으로 반으로 자르지 않는다.

[워터마크]
문구: __________________
PC 위치·농도: __________________
모바일 위치·농도: __________________

[공개 방식]
최종 파일은 __________________에 배포한다.
기존 링크 유지 여부: 유지 / 새 링크 생성

[완료 조건]
1. 모든 산출물의 내용과 페이지 수가 일치한다.
2. 첫 장, 변경 장, 가장 복잡한 장, 마지막 장을 실제 렌더링해 확인한다.
3. 공개 URL이 외부에서 HTTP 200으로 열린다.
4. 작업 로그와 Git 커밋에 관련 파일만 포함된다.

모호한 내용이 있으면 작업 전에 가장 중요한 질문을 최대 3개만 해줘.
질문할 때는 추천 기본값과 그 이유도 함께 알려줘.
```

## 3. 중간 수정 요청 방식

“이상해”, “다시 해줘”만 적으면 AI가 문제의 범위를 추측해야 한다. 아래 4개 항목으로 말하면 재작업이 줄어든다.

```text
[문제가 보이는 위치]
모바일 PDF 42~45페이지

[현재 문제]
중앙 워터마크가 본문과 겹치고 농도가 진하다.

[원하는 결과]
워터마크를 콘텐츠 밖 하단 여백으로 옮기고 작은 서명형으로 바꾼다.

[유지할 것]
현재 글자 크기, 페이지 분할, 공개 링크는 유지한다.
```

## 4. AI가 먼저 물어봤으면 좋았던 질문

작업 전에 아래 질문을 했다면 반복 작업을 크게 줄일 수 있었다.

1. 모바일판은 “원본 보존형”과 “내용 재편집형” 중 어떤 방식인가?
2. 확대 없이 읽는 것이 필수인가? 최소 글자 크기는 얼마인가?
3. 워터마크는 보호가 우선인가, 가독성이 우선인가?
4. 결과 링크는 내 PC에서만 여는 링크인가, 다른 사람도 여는 공개 URL인가?
5. HTML 수정 시 PC·모바일 PDF도 항상 함께 갱신해야 하는가?
6. 기존 공개 URL을 유지해야 하는가?
7. 통계나 차트처럼 요청하지 않은 요소를 제안해도 되는가?
8. 콘텐츠 추가 위치는 어느 설명 전후인가?
9. 완료를 판단할 수 있는 수치와 검수 항목은 무엇인가?

AI에게는 모든 질문을 한꺼번에 요청하기보다, 결과를 바꿀 수 있는 질문만 최대 3개씩 하도록 지시하는 것이 좋다.

## 5. 저장소에 미리 있어야 하는 문서

다음 문서가 있으면 작업이 길어져도 기준을 잃지 않는다.

- `AGENTS.md`: AI가 항상 따라야 하는 작업 규칙
- `docs/AI_작업요청_및_재작업방지_가이드.md`: 사람이 사용하는 요청 템플릿
- `docs/DECISIONS.md`: 비율, 워터마크, 파일명 등 확정된 결정
- `docs/CHANGE_CHECKLIST.md`: HTML·PC·모바일·배포 동기화 체크리스트

## 6. 권장 Git 작업 흐름

### 시작

```text
1. 현재 브랜치와 변경 파일 확인
2. 이번 작업에 포함할 파일 목록 확정
3. 원본 파일과 생성 파일 구분
4. 완료 조건 기록
```

### 작업 중

```text
1. 원본 HTML 수정
2. PC 결과 생성·검수
3. 모바일 결과 생성·검수
4. 선택 페이지의 설명과 페이지 수 수정
5. 전체 문구 파일 갱신
```

### 커밋 전

```text
1. 관련 파일만 명시적으로 스테이징
2. 임시 렌더링 파일이 포함되지 않았는지 확인
3. 스테이징된 파일 목록과 크기 확인
4. 한 커밋이 하나의 사용자 목적을 설명하는지 확인
```

### 배포 후

```text
1. HTML 핵심 문구 확인
2. PC PDF 파일 크기와 페이지 수 확인
3. 모바일 PDF 파일 크기와 페이지 수 확인
4. 공개 URL의 HTTP 200 확인
5. 캐시를 피한 주소로 새 버전 확인
```

## 7. 완료 정의

파일을 생성하거나 커밋한 것만으로 완료가 아니다. 아래 조건을 모두 만족해야 완료다.

- 요청한 내용이 원본에 반영됐다.
- 파생 산출물이 같은 버전으로 생성됐다.
- 모바일에서 확대 없이 읽을 수 있다.
- 워터마크가 내용을 가리지 않는다.
- 선택 페이지의 안내와 실제 파일이 일치한다.
- 공개 링크가 다른 사람에게도 열린다.
- 관련 없는 파일이 커밋되지 않았다.
- 사용자가 다음 수정 때 사용할 문구 파일과 기준 문서가 최신이다.
